# Design: Email Pipeline Redesign (Validate-Repair Architecture)

> Generated by /plan-ceo-review on 2026-04-07 | Branch: main

## Problem Statement

The current `process_emails_openai.py` monolith does classification + scenario matching + drafting + urgency + ownership in a single OpenAI call per email, with group-level KB retrieval (8 emails per RAG query). This produces confidently wrong FM/ready drafts, cross-email KB contamination, FM/bug emails left unread on failure, and no machine-readable business guardrails.

## Vision

"Review dangerous drafts, not ugly drafts." Auto-fix = deterministic string transforms only (no LLM). Business-risk failures → FM/review. Pipeline never blocks on an imperfect draft.

## Implementation Clarifications

**PARTIAL_CONTEXT handling:** `risk_triggers` from `rules_engine.route()` is passed as a parameter to `validators.validate(draft_body, contract, risk_triggers)`. The validator uses risk_triggers to include additional checks (e.g., if `"attachment_present"` in risk_triggers → MEDIUM severity).

**WRONG_SCENARIO vs UNKNOWN_SCENARIO:** `UNKNOWN_SCENARIO` = model scenario ID not in any contract at all (FALLBACK_CONTRACT used). `WRONG_SCENARIO` = pre_route_hint and model scenario ID conflict — recorded in risk_triggers as `"scenario_mismatch"`.

**FORMAT_VIOLATION:** The HIGH check "empty or < 50 chars after LOW fix" emits `FORMAT_VIOLATION`.

**repair_v2 validator_score:** `validators.validate()` ALWAYS runs on repair_v2 output. Both v1 and v2 get a validator_score. Comparison is always valid.

**auto-fix "if missing" match criteria:** Substring search, case-insensitive.
- `"let us know if you have any questions" in draft_body.lower()`
- `draft_body.lower().startswith("dear ")`
- `"best regards" in draft_body.lower()`

**kb_text in repair_v2:** Same `kb_text` variable from the per-email RAG call — not re-fetched.

**Bug failure draft body:**
```python
f"[REVIEW NEEDED: Bug ticket creation failed — {error_message}]\n\n"
f"Dear {customer_name},\n\n"
f"Thank you for reaching out about the issue with {issue_summary}. "
f"We have received your report and will follow up shortly.\n\n"
f"Let us know if you have any questions,\n\nBest regards,"
```

**rag.get_relevant_context() empty fallback:** Handled inside rag.py line 85: `return rules_text`. No change needed.

**upsert key:** Composite key = (Date, review_reason_code). Row update or append.

**rules_engine pure function:** Receives pre-processed normalized email dict — no Gmail API calls inside rules_engine.

## Pipeline (Correct Sequencing)

**Critical:** `scenario_contracts.select()` runs AFTER the OpenAI v1 call — it needs `model_scenario_id` from model output. `rules_engine.route()` provides `pre_route_hint` before the LLM call as a hint only.

```
normalize_thread(thread_data, email_meta)
  = _compact_summary() renamed, stays in orchestrator as private helper

rules_engine.route(email)
  returns: {sender_type, is_bug, risk_triggers, pre_route_hint}

  ┌── ALREADY_REPLIED: if email.has_support_reply → FM/review (skip LLM)
  ├── PARTIAL_CONTEXT: if email.has_attachments and no visible attachments
  │       → adds "attachment_present" to risk_triggers
  └── is_bug: if bug heuristic fires → is_bug = True

if is_bug:
    apply_label(FM/bug) immediately   ← BEFORE ticket creation
    create_bug_ticket()
        success → state_module.save_email(review_status=bug)
        FAIL    → FM/review draft (bug acknowledgment text), FM/bug label kept
                  state_module.save_email(review_status=bug_failed,
                                          review_reason_code=BUG_TICKET_FAILED)

else:
    rag.get_relevant_context(email_text=email['latest_message'])
        [BM25 per-email; empty → returns rules_text fallback inside rag.py:85]

    call_openai(draft_v1, schema: {scenario, scenario_confidence, draft_body, classification})
        JSONDecodeError or missing fields → FM/review, review_reason_code=AI_ERROR

    scenario_contracts.select(pre_route_hint, model_scenario_id=output["scenario"])
        conflict → log warning, record "scenario_mismatch" in risk_triggers, use model_scenario_id
        no match → FALLBACK_CONTRACT, review_reason_code=UNKNOWN_SCENARIO

    if confidence < CONFIDENCE_THRESHOLD (0.7):
        → FM/review, review_reason_code=LOW_CONFIDENCE   [skip validator]

    else:
        validators.validate(draft_body, contract, risk_triggers)
            returns: {severity, issues, validator_score: float 0.0–1.0}

            LOW  → auto-fix (string transforms) → FM/ready

            MED  → repair_v2:
                     sys_prompt = build_classify_prompt(email, kb_text)[0]
                     sys_prompt += "\n\n[VALIDATION ERRORS:\n{issues}\nFix all.]"
                     call_openai(repair_v2, max_tokens=2000)
                     validators.validate(repair_v2, contract, risk_triggers)  [always runs]
                     PASS → FM/ready
                     FAIL → FM/review, keep draft with higher validator_score
                            repair_attempted=True

            HIGH → FM/review, preserve draft_v1

    final_label → state_module.save_email(validator_score, repair_attempted, review_reason_code)
```

**confidence < 0.7 is ONLY a pre-validator gate. HIGH severity = policy/ownership violations from validator.**

## Architecture — New Modules

- `src/api/rules_engine.py` — deterministic pre-routing. Runs BEFORE LLM call.
- `src/api/validators.py` — post-LLM validation. Returns normalized validator_score. Performs LOW auto-fix.
- `src/api/scenario_contracts.py` — loads JSON contracts from `knowledge/contracts/` at startup, fail-fast on invalid, provides FALLBACK_CONTRACT.

## review_reason_code Enum

| Code | Assigned by | Trigger |
|---|---|---|
| `LOW_CONFIDENCE` | orchestrator | scenario_confidence < 0.7 |
| `ALREADY_REPLIED` | rules_engine | has_support_reply = True |
| `PARTIAL_CONTEXT` | rules_engine | has_attachments + no visible attachments |
| `WRONG_OWNERSHIP` | validators | Draft claims wrong authority |
| `MISSING_REQUIRED_FACT` | validators | required_facts not addressed |
| `FORBIDDEN_PROMISE` | validators | forbidden_promises content in draft |
| `FORMAT_VIOLATION` | validators HIGH | Draft empty or < 50 chars after auto-fix |
| `WRONG_SCENARIO` | scenario_contracts.select | pre_route_hint vs model conflict |
| `AI_ERROR` | orchestrator | JSONDecodeError or missing fields |
| `BUG_TICKET_FAILED` | orchestrator | create_bug_ticket() raised |
| `UNKNOWN_SCENARIO` | scenario_contracts.select | No matching contract |

## Validator Severity Tiers

| Check | Severity | Auto-fix? |
|---|---|---|
| Markdown in body | LOW | Yes — regex strip |
| Missing salutation | LOW | Yes — prepend |
| Missing "Let us know..." closing | LOW | Yes — append |
| Missing "Best regards," | LOW | Yes — append |
| pre_route_hint vs model scenario conflict | MEDIUM | No — repair |
| required_fact not addressed | MEDIUM | No — repair |
| PARTIAL_CONTEXT risk trigger | MEDIUM | No — repair |
| Ownership mismatch | HIGH | No — downgrade |
| Forbidden promise | HIGH | No — downgrade |
| Empty/short draft (<50 chars) | HIGH | No — downgrade |

**validator_score:** float = passed_checks / total_checks. Normalized. Comparable between v1 and repair_v2.

## scenario_confidence Schema

```json
{ "scenario_confidence": 0.85 }
```
Float 0.0–1.0. Model instructed: < 0.5 for ambiguous, > 0.8 for clear match.
Known limitation: LLM confidence is not perfectly calibrated. Tune 0.7 threshold after first week of prod data.

```python
sys_prompt, usr_prompt = build_classify_prompt(email, kb_text)
```

```python
contracts = scenario_contracts.load_all()  # sys.exit(1) on schema error
scenario_ids = extract_scenario_ids_from_kb(knowledge.load_scenarios())
```

## Bug Dedup

```python
seen_thread_ids = set()
deduped_bug_emails = []
for item in all_bug_emails:
    tid = item["email"].get("thread_id")
    if tid not in seen_thread_ids:
        seen_thread_ids.add(tid)
        deduped_bug_emails.append(item)
```
